# 商品秒杀平台 - 系统概述与架构设计

## 1. 项目背景与目标

随着电商平台业务规模的扩大，秒杀活动已成为吸引用户流量、提升平台活跃度的核心营销手段。然而秒杀场景对系统的冲击远超普通业务：瞬时流量峰值可达日常流量的数十倍，任何细微的性能瓶颈都会被急剧放大，引发雪崩效应。

本项目旨在构建一套**高可用、高性能、可水平扩展**的商品秒杀平台，具备以下核心能力：

- **瞬时高并发处理**：支撑万级 QPS 的秒杀请求冲击
- **零超卖保障**：库存扣减严格准确，不出现负数库存
- **快速响应体验**：用户从发起请求到获得结果，平均耗时控制在 50ms 以内
- **服务高可用**：任意单个节点故障不影响整体服务可用性

## 5. 核心设计理念

### 5.1 分层漏斗：每一层尽可能提前过滤无效请求

```
原始请求 (100,000 QPS)
    │
    ▼
[第1层] Nginx limit_req 限流  ──→ 过滤 50%，剩余 50,000 QPS
    │
    ▼
[第2层] Gateway 路由 + 参数校验 ──→ 过滤 30%，剩余 10,000 QPS
    │
    ▼
[第3层] Redis Lua 原子扣库存 ──→ 过滤 99%，剩余约 100 QPS
    │
    ▼
[第4层] RabbitMQ 消息队列削峰 ──→ 匀速消费，~100 QPS 写入
    │
    ▼
[第5层] MySQL 持久化 ──→ 最终订单落库
```

### 5.2 四大设计原则

| 设计目标 | 核心手段 | 说明 |
|---------|---------|------|
| **高性能** | Redis 内存操作 + MQ 异步削峰 | Redis Lua < 0.1ms，异步下单避免 MySQL 阻塞 |
| **强一致性** | Redis Lua 原子扣减 + 乐观锁兜底 + 唯一索引防重 | 三层保障实现零超卖 |
| **高可用** | 多实例部署 + 无状态服务 + 中间件集群 | 任意节点故障不影响整体 |
| **可扩展** | 微服务独立部署 + Nacos 动态配置 | 新增实例无需修改代码 |

### 5.3 数据一致性策略

Redis 与 MySQL 双写一致性采用**最终一致性 + 补偿机制**：

1. **优先 Redis**：用户请求到达后，优先在 Redis 中完成库存预扣减（Lua 原子操作）
2. **异步 MySQL 确认**：Redis 扣减成功后，发送 MQ 消息，由 Consumer 异步完成 MySQL 写入
3. **失败补偿**：若 MySQL 写入失败，则回滚 Redis 库存，并记录异常
4. **定时对账**：每小时定时任务比对 Redis 库存与 MySQL 库存，发现差异自动修正
