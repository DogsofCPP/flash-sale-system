# 功能模块设计

## 1. 模块划分总览

系统按业务边界划分为 5 个独立微服务，各服务职责单一，通过 HTTP/gRPC 进行服务间通信：

| 服务名称 | 端口 | 核心职责 | 持久化数据库 | 关键机制 |
|---------|------|---------|------------|---------|
| gateway | 8080 | 统一入口，路由分发，鉴权过滤 | 无 | StripPrefix |
| user-service | 8081 / 8082 | 用户注册登录，认证鉴权 | flash_sale_user | 读写分离 AOP |
| product-service | 8083 | 商品 CRUD，ES 全文搜索 | flash_sale_product | 多级缓存 |
| seckill-service | 8086 / 实例2 | 秒杀核心，库存扣减 | flash_sale | Lua 原子扣减 + 限流 |
| order-service | 8085 | 订单创建/取消/超时，MQ 消费 | flash_sale_order | 定时任务调度 |

## 3. 库存管理

### 3.1 库存操作策略

系统采用**预扣减 + 异步确认 + 最终兜底**三层库存保障体系：

| 层级 | 操作位置 | 手段 | 目的 |
|------|---------|------|------|
| 第一层 | Redis | Lua 脚本原子扣减 | 快速响应，拦截 99% 无效请求 |
| 第二层 | RabbitMQ | 异步消息消费 | 削峰填谷，平滑写入 MySQL |
| 第三层 | MySQL | 乐观锁 UPDATE WHERE | 最终一致性兜底 |

### 3.2 Lua 脚本实现

**扣减库存脚本 `decrease_stock.lua`**

```lua
-- Key[1]: 库存 key = flash:seckill:stock:{activityId}:{productId}
-- Key[2]: 用户购买记录 key = flash:seckill:user:{activityId}:{productId}:{userId}
-- ARGV[1]: 本次购买数量
-- ARGV[2]: 限购上限

-- 第一步：检查用户是否已达购买上限
local user_bought = redis.call('GET', KEYS[2])
if user_bought and tonumber(user_bought) >= tonumber(ARGV[2]) then
    return -1  -- -1 表示已达限购上限
end

-- 第二步：检查库存是否充足
local stock = redis.call('GET', KEYS[1])
if not stock then
    return -2  -- -2 表示 Redis 中库存未初始化
end
if tonumber(stock) < tonumber(ARGV[1]) then
    return -3  -- -3 表示库存不足
end

-- 第三步：原子扣减库存 + 更新用户购买计数
redis.call('DECRBY', KEYS[1], ARGV[1])
redis.call('SET', KEYS[2], tonumber(user_bought or 0) + tonumber(ARGV[1]))

-- 第四步：返回扣减后的剩余库存
local remain = redis.call('GET', KEYS[1])
return tonumber(remain)
```

**回滚库存脚本 `increase_stock.lua`**

```lua
-- Key[1]: 库存 key
-- ARGV[1]: 回滚数量

local stock = redis.call('GET', KEYS[1])
if not stock then
    return -1
end
return redis.call('INCRBY', KEYS[1], tonumber(ARGV[1]))
```

### 3.3 库存预热

活动开始前，由定时任务将 MySQL 中的库存数据同步到 Redis：

```
定时任务（活动开始前5分钟触发）
    │
    ▼
读取 seckill_product 表，获取 stock 和 limit_per_user
    │
    ▼
写入 Redis:
  SET flash:seckill:stock:{activityId}:{productId} = stock
  EXPIRE 活动结束时间 + 1小时
    │
    ▼
记录预热日志
```

## 4. 订单生命周期

### 4.1 状态流转

```
  ┌─────────────────────────────────────────────────────┐
  │                                                     │
  │      ┌──────────┐    用户支付    ┌──────────┐       │
  │      │  待支付   │ ───────────→ │  已支付   │       │
  │      └────┬─────┘              └─────┬────┘       │
  │           │                         │             │
  │    用户取消│                         │ 申请退款     │
  │           │                         ▼             │
  │           ▼                   ┌──────────┐        │
  │      ┌──────────┐             │  已退款   │        │
  │      │  已取消   │             └──────────┘        │
  │      └──────────┘                                 │
  │           │                                       │
  │    定时扫描│（15分钟超时）                          │
  │           ▼                                       │
  │      ┌──────────┐                                  │
  │      │  已超时   │                                  │
  │      └──────────┘                                  │
  │                                                     │
  └─────────────────────────────────────────────────────┘
```

| 状态码 | 状态名 | 触发条件 | 后续动作 |
|--------|--------|---------|---------|
| 0 | 待支付 | 订单创建 | 等待支付 / 等待超时 |
| 1 | 已支付 | 用户付款 | 可申请退款 |
| 2 | 已取消 | 用户主动取消 | 触发库存回滚 |
| 3 | 已退款 | 退款完成 | 触发库存回滚 |
| 4 | 已超时 | 15分钟未支付 | 触发库存回滚 |

### 4.2 超时订单自动关闭

Order-Service 每 30 秒扫描一次待支付订单：

```sql
SELECT * FROM t_order
WHERE status = 0
  AND expire_time <= NOW()
LIMIT 100
```

对每条过期订单执行：

1. 更新 `status = 4`（已超时）
2. 调用 Seckill-Service 回滚 Redis 库存（`INCRBY flash:seckill:stock:{activityId}:{productId} quantity`）
3. 记录 `stock_flow`（type=2 回滚）
4. 更新 `seckill_order.status = 3`（超时）

## 5. 分布式事务

### 5.1 TCC 事务模式

秒杀订单创建涉及 Seckill-Service 和 Order-Service 两个服务，采用 TCC 模式保障分布式一致性：

| 阶段 | Seckill-Service（扣库存） | Order-Service（建订单） |
|------|-------------------------|----------------------|
| **Try** | Redis Lua 预扣库存 | 创建订单（status=0） |
| **Confirm** | 确认扣减，更新 `sold_count` | 更新订单为已创建（status=1） |
| **Cancel** | Redis 回滚库存 +1 | 标记订单已取消（status=2） |

### 5.2 MQ 可靠消息

```
1. Seckill-Service 执行 Redis Lua 扣减
2. 发送 RabbitMQ 半消息（事务消息）
3. MQ Broker 回执确认
4. Order-Service 消费消息，创建 MySQL 订单
5. 若 Order-Service 消费失败 → NACK 消息 → 重试
   若 Seckill-Service 未收到回执 → 触发 Cancel 回滚
```

## 6. 定时任务设计

| 服务 | 任务名称 | 执行周期 | 核心逻辑 |
|------|---------|---------|---------|
| seckill-service | 库存预热 | 每分钟 | 扫描 start_time 未来5分钟的活动，同步库存到 Redis |
| seckill-service | 活动状态调度 | 每分钟 | 更新活动状态：未开始→进行中→已结束 |
| seckill-service | 库存对账 | 每小时 | 比对 Redis 与 MySQL 库存，差异则修正 |
| order-service | 超时订单关闭 | 每30秒 | 扫描 expire_time ≤ NOW() 且 status=0 的订单 |
| order-service | 异常订单处理 | 每小时 | 重试多次失败的 MQ 消费消息 |

## 7. API 网关路由配置

网关使用 Spring Cloud Gateway 实现统一入口，通过路由规则将请求转发到对应微服务：

```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: user-service
          uri: lb://user-service
          predicates:
            - Path=/api/users/**
          filters:
            - StripPrefix=1

        - id: product-service
          uri: lb://product-service
          predicates:
            - Path=/api/products/**
          filters:
            - StripPrefix=1

        - id: seckill-service
          uri: lb://seckill-service
          predicates:
            - Path=/api/seckill/**
```

## 8. 模块间通信

### 8.1 HTTP 同步调用

| 调用方 | 被调用方 | 接口 | 说明 |
|-------|---------|------|------|
| seckill-service | order-service | POST /internal/orders | 创建正式订单 |
| order-service | seckill-service | POST /internal/seckill/rollback | 通知库存回滚 |
| gateway | 各微服务 | /api/** | 外部请求统一入口 |

### 8.2 MQ 异步消息

| Topic | 生产者 | 消费者 | 消息内容 |
|-------|-------|-------|---------|
| flash.seckill.order | seckill-service | order-service | 秒杀订单创建消息 |
| flash.order.cancel | order-service | seckill-service | 订单取消/超时通知 |
| flash.stock.rollback | order-service | seckill-service | 库存回滚通知 |

### 8.3 服务注册与发现

所有微服务启动时向 Nacos 注册，服务间通过服务名而非 IP:Port 进行调用：

```
seckill-service 调用 order-service:
  → Nacos 查询 order-service 的可用实例列表
  → Gateway 负载均衡选择一个实例
  → 发起 HTTP 调用
```
