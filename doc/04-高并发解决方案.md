# 高并发解决方案

## 1. 问题背景与挑战

秒杀场景的核心矛盾是：**海量用户同时争抢有限商品**。与传统业务不同，秒杀系统的流量特征呈现"瞬时峰值"——在活动开始的那几秒内，请求量可能从日常的几十 QPS 暴涨至上万甚至十万 QPS。

如果不加控制地直接涌入数据库，所有请求都会阻塞在库存行锁上，导致：

- 数据库连接池耗尽，影响其他业务
- 大量请求超时，用户体验极差
- 数据库 CPU 飙升，引发雪崩

本系统的解决思路是：**在流量到达数据库之前，通过多层手段逐步消化无效请求**。

## 2. 整体解决方案架构

```
┌──────────────────────────────────────────────────────────────────────┐
│                        请求处理全链路                                  │
│                                                                      │
│  100,000 QPS ──→ Nginx限流 ──→ Gateway限流 ──→ Redis预扣 ──→ MQ削峰 ──→ MySQL │
│                     -50%       -60%          -99%       平滑化      最终落库 │
│                                                                      │
│  经过五层处理，真正打到数据库的只有约 100 QPS                           │
└──────────────────────────────────────────────────────────────────────┘
```

## 3. 接入层限流（Nginx）

### 3.1 基于 IP 的请求速率限制

Nginx 使用 `limit_req_zone` 模块实现令牌桶限流：

```nginx
# 定义限流规则：基于客户端IP，每秒最多10个请求
limit_req_zone $binary_remote_addr zone=flash_sale_ip:10m rate=10r/s;

server {
    listen 80;

    location /api/seckill/seckill {
        # 突发容量20，超过则返回503
        limit_req zone=flash_sale_ip burst=20 nodelay;
        proxy_pass http://gateway;
    }
}
```

### 3.2 并发连接数限制

```nginx
# 限制单个 IP 最大并发连接数
limit_conn perip 5;
# 限制服务器总连接数
limit_conn addr 1000;
```

### 4.2 热点接口独立限流

对于秒杀核心接口，使用 Guava RateLimiter 实现更精细的控制：

```java
@Component
public class SeckillRateLimiter {

    private final Map<Long, RateLimiter> limiterMap = new ConcurrentHashMap<>();

    @Autowired
    private SeckillActivityMapper activityMapper;

    /**
     * 根据活动ID获取对应的限流器
     * 不同活动独立限流，避免一个活动影响另一个
     */
    public RateLimiter getLimiter(Long activityId) {
        return limiterMap.computeIfAbsent(activityId, id -> {
            SeckillActivity activity = activityMapper.selectById(id);
            // 每个活动每秒允许100个请求
            return RateLimiter.create(
                activity != null ? activity.getQpsLimit() : 100.0
            );
        });
    }

    /**
     * 尝试获取令牌，超时则拒绝
     * @return true 表示允许通过，false 表示被限流
     */
    public boolean tryAcquire(Long activityId, long timeout, TimeUnit unit) {
        RateLimiter limiter = getLimiter(activityId);
        return limiter.tryAcquire(timeout, unit);
    }
}
```

## 5. Redis 缓存策略

### 5.1 多级缓存架构

```
请求 ──→ Nginx静态缓存 ──→ Gateway本地缓存 ──→ Redis分布式缓存 ──→ MySQL
        (几秒TTL)        (5秒TTL)          (活动维度)              (最终DB)
```

### 5.2 热点数据缓存

商品详情、秒杀活动信息等读多写少的数据放入 Redis：

```java
@Service
public class ProductCacheService {

    @Autowired
    private RedisTemplate<String, Object> redisTemplate;

    private static final String PRODUCT_KEY = "flash:product:detail:";
    private static final Duration CACHE_TTL = Duration.ofMinutes(10);

    public Product getProduct(Long productId) {
        String key = PRODUCT_KEY + productId;

        // 先查缓存
        Product product = (Product) redisTemplate.opsForValue().get(key);
        if (product != null) {
            return product;
        }

        // 缓存未命中，查数据库
        product = productMapper.selectById(productId);

        // 回填缓存
        if (product != null) {
            redisTemplate.opsForValue().set(key, product, CACHE_TTL);
        }

        return product;
    }

    // 商品更新时清除缓存（Cache-Aside 模式）
    public void evictCache(Long productId) {
        redisTemplate.delete(PRODUCT_KEY + productId);
    }
}
```

### 5.3 缓存三大问题与应对

| 问题 | 描述 | 解决方案 |
|------|------|---------|
| **缓存穿透** | 查询不存在的数据，每次都穿透到 DB | 布隆过滤器 + 空值缓存（短TTL） |
| **缓存击穿** | 热点 key 过期瞬间，大量请求击穿到 DB | 互斥锁（SETNX）+ 永不过期策略 |
| **缓存雪崩** | 大量缓存同时过期 | 过期时间加随机偏移量 + 持久化备份 |

## 6. Redis 原子扣减（核心）

### 6.1 为什么不用悲观锁

传统做法是 `SELECT ... FOR UPDATE` 锁定行，然后 `UPDATE stock - 1`。在秒杀场景下：

- 大量请求排队等待锁
- 锁持有时间长（网络往返 + 业务处理）
- 并发越高，等待队列越长
- MySQL 连接池快速耗尽

### 6.2 为什么选择 Lua + Redis

Redis 执行 Lua 脚本具有以下特性：

- **单线程执行**：Redis 本身是单线程，不存在并发竞争
- **原子性保证**：Lua 脚本作为整体执行，不会被中断
- **本地执行**：所有操作在 Redis 服务端完成，无网络往返开销
- **O(1) 时间复杂度**：库存检查和扣减均为常数时间

### 6.3 完整 Lua 扣减脚本

```lua
-- ============================================
-- 秒杀库存原子扣减脚本
-- Key[1]: 库存计数器
-- Key[2]: 用户购买记录
-- ARGV[1]: 购买数量
-- ARGV[2]: 限购上限
-- ARGV[3]: 用户已购数量（由应用层传入，避免二次查询）
-- ============================================

-- 第一步：检查用户已购数量是否超限
if tonumber(ARGV[3]) >= tonumber(ARGV[2]) then
    return {-4, 0}  -- 状态码-4表示超出限购
end

-- 第二步：读取当前库存
local currentStock = tonumber(redis.call('GET', KEYS[1]) or '0')

-- 第三步：检查库存是否足够
if currentStock < tonumber(ARGV[1]) then
    return {-3, currentStock}  -- 状态码-3表示库存不足
end

-- 第四步：执行原子扣减
local newStock = redis.call('DECRBY', KEYS[1], ARGV[1])

-- 第五步：更新用户购买记录（使用 INCRBY 保证原子递增）
local newUserBought = redis.call('INCRBY', KEYS[2], ARGV[1])

-- 第六步：返回结果
-- 返回 {状态码, 扣减后剩余库存}
-- 状态码 1 表示扣减成功
return {1, newStock}
```

### 6.4 Java 调用层

```java
@Service
public class RedisStockService {

    private static final DefaultRedisScript<List> DEDUCT_SCRIPT;

    static {
        DEDUCT_SCRIPT = new DefaultRedisScript<>();
        DEDUCT_SCRIPT.setScriptText(loadLuaScript("decrease_stock.lua"));
        DEDUCT_SCRIPT.setResultType(List.class);
    }

    @Autowired
    private RedisTemplate<String, String> redisTemplate;

    public SeckillResult deductStock(Long activityId, Long productId,
                                     Long userId, Integer quantity) {
        String stockKey = String.format("flash:seckill:stock:%d:%d", activityId, productId);
        String userKey = String.format("flash:seckill:user:%d:%d:%d", activityId, productId, userId);

        List<Long> result = redisTemplate.execute(
            DEDUCT_SCRIPT,
            Arrays.asList(stockKey, userKey),
            String.valueOf(quantity),
            "1",  // 限购1件
            "0"   // 用户已购0件
        );

        long statusCode = result.get(0);
        long remainStock = result.get(1);

        return SeckillResult.builder()
            .code(statusCode)
            .remainStock(remainStock)
            .build();
    }
}
```

## 7. 消息队列削峰（RabbitMQ）

### 7.1 异步处理的价值

即使 Redis 扣减成功，也不能同步写 MySQL：

- MySQL 单库 TPS 上限约 1000-2000（InnoDB 行锁）
- 同步写入会阻塞用户请求，用户等待时间长
- 瞬时高峰会导致连接池耗尽

RabbitMQ 的作用是将**尖峰流量**转化为**平稳流量**：

```
瞬时高峰: 10,000 QPS (Redis扣减)
    │
    │ 全部写入 MQ
    ▼
RabbitMQ 队列缓冲
    │
    │ 匀速消费 100 QPS
    ▼
MySQL 平稳落库
```

### 7.3 消费者幂等性保障

MQ 消费天然存在重复投递问题，必须保证幂等：

```java
@RabbitListener(queues = "flash.seckill.order.queue", concurrency = "3")
public class SeckillOrderConsumer {

    @Autowired
    private OrderMapper orderMapper;

    @RabbitHandler
    public void handleOrder(SeckillOrderMessage message, Channel channel,
                            @Header(AmqpHeaders.DELIVERY_TAG) long tag) {
        String orderNo = message.getOrderNo();

        try {
            // 幂等检查：订单号已存在则直接确认，跳过处理
            if (orderMapper.countByOrderNo(orderNo) > 0) {
                log.info("重复消息检测到，跳过: {}", orderNo);
                channel.basicAck(tag, false);
                return;
            }

            // 执行订单创建逻辑
            orderService.createSeckillOrder(message);

            // 手动确认消息
            channel.basicAck(tag, false);

        } catch (OptimisticLockingFailureException e) {
            // 乐观锁冲突，重试
            channel.basicNack(tag, false, true);
        } catch (DuplicateKeyException e) {
            // 重复键异常（数据库唯一约束），直接确认
            channel.basicAck(tag, false);
        } catch (Exception e) {
            log.error("订单处理异常: {}", orderNo, e);
            // 消费失败，重新入队重试
            channel.basicNack(tag, false, true);
        }
    }
}
```

### 7.4 死信队列兜底

```java
    return BindingBuilder.bind(deadLetterQueue())
```

## 8. MySQL 持久化兜底

### 8.1 乐观锁扣减 SQL

Redis 是第一道防线，MySQL 是最终兜底。Order-Service 消费消息后：

```sql
-- 乐观锁扣减：只有库存充足且版本匹配时才扣减成功
UPDATE seckill_product
SET sold_count = sold_count + #{quantity},
    updated_at = NOW()
WHERE id = #{spId}
  AND (stock - sold_count) >= #{quantity}
  AND #{quantity} > 0
```

### 8.2 回滚库存

订单取消、超时或 MQ 消费失败时回滚：

```sql
UPDATE seckill_product
SET sold_count = sold_count - #{quantity}
WHERE id = #{spId}
  AND sold_count >= #{quantity}
```

## 9. 分布式锁

### 9.1 典型使用场景

| 场景 | 锁 Key | 说明 |
|------|--------|------|
| 库存预热 | `lock:stock:warmup:{activityId}` | 防止多实例同时预热 |
| 活动状态更新 | `lock:activity:update:{activityId}` | 防止并发更新活动状态 |
| 库存对账 | `lock:stock:reconcile:{activityId}` | 单实例执行对账任务 |

### 9.2 Redis 分布式锁实现

```java
@Component
public class DistributedLockService {

    @Autowired
    private RedisTemplate<String, String> redisTemplate;

    private static final String LOCK_PREFIX = "lock:";

    /**
     * 尝试获取分布式锁
     * @param lockKey 锁的键
     * @param expireSeconds 过期时间（秒）
     * @return 锁的持有标识，用于安全释放
     */
    public String tryLock(String lockKey, long expireSeconds) {
        String lockValue = UUID.randomUUID().toString();
        String fullKey = LOCK_PREFIX + lockKey;

        Boolean success = redisTemplate.opsForValue()
            .setIfAbsent(fullKey, lockValue, expireSeconds, TimeUnit.SECONDS);

        return success != null && success ? lockValue : null;
    }

    /**
     * 释放锁（Lua 脚本保证原子性：只释放自己持有的锁）
     */
    public void unlock(String lockKey, String lockValue) {
        String fullKey = LOCK_PREFIX + lockKey;
        String script =
            "if redis.call('GET', KEYS[1]) == ARGV[1] then " +
            "  return redis.call('DEL', KEYS[1]) " +
            "else return 0 end";

        redisTemplate.execute(
            new DefaultRedisScript<>(script, Long.class),
            Collections.singletonList(fullKey),
            lockValue
        );
    }
}
```
